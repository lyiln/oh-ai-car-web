# 基础需求补全与验收计划

## 使用方式

本计划以《五人三天详细任务计划书》的 A-E 分工为唯一职责基线。原计划解决的是中期前的最小实验和展示；本文件补充课程五项基本需求的最终验收，不重新分配成员，也不把 Web 控制端纳入基础任务。

所有模块当前状态均为“待核验”。只有负责人填入可定位证据并由组长复核后，才可改为“已验证”。

## 中期展示与最终验收

| 阶段 | 目标 | 可接受证据 | 不足以证明的事项 |
| --- | --- | --- | --- |
| 中期检查 | 证明每位成员已启动基础任务，并能说明下一步 | 截图、日志、仿真、静态推理、阶段录屏、失败记录 | 真车运行、跨模块闭环、云平台/CI/CD 已完成、系统安全已验证 |
| 最终基础验收 | 证明五项基本需求均有可运行实现和负责人 | 代码、启动命令、运行日志、视频、个人讲解、异常记录 | 仅有 PPT、仅有参考源码、仅有他人操作结果 |
| 进阶项申报 | 在基础五项已验证后展示系统优化 | 教师认可、对比场景、量化或行为证据、局限说明 | 以未完成基础任务替代加分项 |

## 五项基本需求补全

### ROS2 核心开发与分布式通信

- 主责：成员 C；协作：成员 E，成员 A。
- 原计划已覆盖：A1 雷达、Docker、`/scan` 获取、避障节点、`/cmd_vel`、`rqt_graph`。
- 待补齐：明确节点发布者和订阅者、消息类型、启动顺序、跨节点数据流，以及视觉节点接入后的话题关系。
- 最终验收：ROS2 工作空间可构建；至少一条 `传感器/视觉节点 -> 应用节点 -> /cmd_vel` 或等价话题链路可展示；保存 Launch、`ros2 topic`、`rqt_graph`、终端日志和异常记录。

### SLAM 建图与自主导航

- 主责：成员 A；协作：成员 C。
- 原计划已覆盖：仿真建图、地图保存、导航准备，以及中期后实车导航和多点导航。
- 待补齐：区分仿真与实车证据，记录地图质量、初始位姿、目标点、路径结果和失败原因。
- 最终验收：至少完成仿真建图和导航；实车条件允许时完成小范围建图及单点导航。保存 `pgm/yaml`、RViz 截图、启动命令、路径视频和异常记录。

### AI 边缘推理与多传感器融合

- 主责：成员 E；协作：成员 C。
- 原计划已覆盖：Astra 相机、图像采集、YOLO 标注和基础推理。
- 待补齐：训练或明确采用的模型版本、边缘端推理、RGB 检测结果与深度距离的对应关系、输出到 ROS2 追踪/控制节点的数据链路。
- 最终验收：保留数据样本、标注、模型/权重说明、边缘推理日志、RGB 与深度结果、目标位置输出和 ROS2 话题记录。仅预训练静态图推理只能作为中期证据。

### 云平台集成与 CI/CD 流水线

- 主责：成员 B；协作：成员 A。
- 原计划已覆盖：传感器采集、WiFi/NB、云平台记录、仓库规范和提交物管理。
- 待补齐：云平台与 CI/CD 是两项独立证据，不能以日报或仓库截图替代流水线。
- 最终验收：至少一条传感器或网络数据上云并保留云端记录；至少一条可重复执行的 CI/CD 流水线完成构建、测试和版本化制品发布或测试环境部署，并保留运行日志、失败处理、制品链接和部署结果。

### 鸿蒙 APP 控制小车运动

- 主责：成员 D；协作：成员 C。
- 原计划已覆盖：基础页面、摇杆/按钮交互、TCP 指令发送；中期后 APP 与 ROS2 闭环、异常处理和编码完善。
- 待补齐：DevEco/SDK 环境、工程可运行证据、连接失败处理、Stop/释放行为和 APP 到 ROS2/小车的控制记录。
- 最终验收：展示网络配置、遥控或麦克纳姆轮入口、按钮/摇杆、至少 2-3 类 TCP 指令及停止行为；保存页面截图、触摸日志、TCP/ROS2 日志和安全测试记录。当前 Web 控制端不替代该项。

## 集成演示规则

最终展示使用“自主巡检与人工控制”叙事：A 展示地图/导航，C 展示雷达和 ROS2 通信，E 展示视觉与融合，B 展示云端与流水线，D 展示鸿蒙 APP 控制。每个环节必须标注为“已联动”“并列展示”或“待验证”；不得把并列展示叙述为自动闭环。

## 收口顺序

1. 每位成员先在证据矩阵填入仓库、命令、截图/日志和当前风险。
2. A 按五项需求检查是否同时具备中期证据与最终证据计划。
3. B 建立最终提交目录，汇总日报、视频、地图、模型、日志和版本信息。
4. 真车可用后，按安全要求完成低风险验证并回填结果；失败同样保留记录。
5. 五项全部“已验证”后，才将 Web 控制端作为进阶候选提交给教师评估。

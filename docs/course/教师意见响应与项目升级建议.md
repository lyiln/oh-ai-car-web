# 教师意见响应与项目升级建议

## 项目特色定位

项目由“车牌识别巡检”升级为“社区停车问题发现—人工复核—车辆协作—上门留证—物业通知”的主动处置闭环。核心痛点是物业人员发现乱停车后仍需人工查询车主、定位住户并往返通知；本项目让小车承担发现、取证和到户留证，物业保留最终确认权。

## 对教师建议的响应

### 1. 提高 AI 使用率

- Jetson 边缘端负责车辆检测、车牌 OCR、禁停 ROI 判断及证据筛选。
- 云端 AI 根据脱敏后的违规类型、位置、置信度生成处置建议和通知话术；服务不可用时使用规则模板，演示不依赖外网。
- AI 不直接控制底盘。只有规则校验通过、命中登记私家车、存在一层目的地且物业人工确认后，系统才允许分配导航任务。
- 答辩 PPT 可用“端—边—云—Web”数据流、处置时间轴和失败降级演示突出 AI 的实际作用。

### 2. 深挖场景痛点

区别于常见的巡逻、避障项目，本组聚焦社区乱停车处置断点：识别结果与物业工作没有自动衔接。特色功能不是单纯“看见车”，而是将证据、白名单、住户一层入口、Nav2 和物业确认串成可审计闭环。

### 3. 多车协作

- 巡检车发现问题后，平台优先选择同地图、在线、空闲且电量不低于 20% 的其他车辆执行上门任务。
- 没有可用车辆时，允许源巡检车作为单车降级方案；ROS 调度器仍须先安全暂停巡检并保存断点。
- 当前采用确定性规则调度，不宣称实现强化学习或复杂全局路径优化。

### 4. 端边云协同与现实设备联动

| 层次 | 职责 |
| --- | --- |
| 端 | 相机、雷达、底盘、Nav2、急停和零速度确认 |
| 边 | YOLO/OCR、深度融合、证据筛选、断网缓存 |
| 云 | 车辆/住户资料、任务状态、规则调度、AI 文本、报表与审计 |
| Web | 人工复核、目的地配置、任务确认、实时监控与异常接管 |

系统联动物业白名单、楼栋一层入口点、社区地图和摄像头证据。MVP 不控制门禁，不进入楼内，不处理电梯或自动拨号。

### 5. 项目与风险管理

- 每天联调前完成相机、雷达、底盘、Nav2、网络五项冒烟测试。
- 保留设备 API 模拟器、单车降级、AI 模板降级、Gazebo/Nav2 仿真和录制证据输入。
- 真车演示前冻结功能，只修复阻断问题；现场配置物理安全员、急停范围和低速上限。
- 每项证据明确负责人并保存命令、日志、截图和录像。

## 答辩表述边界

- **已实现**：Web 数据模型、目的地配置、人工确认、规则分配、AI 降级、设备接口、幂等事件与零速度门槛。
- **接口已实现**：ROS 调度器领取上门任务、上报导航/到达/留证/完成事件。
- **仿真已验证**：仅在实际完成 Gazebo/Nav2 联调并保存证据后使用。
- **真车已验证**：仅在完成真实地图、导航、停车和安全接管实验并保存记录后使用。

Web 自动化测试和模拟设备不能替代真车验证，也不能证明真实 OCR 精度或车辆已经物理停止。

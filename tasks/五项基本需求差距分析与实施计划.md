# 五项基本需求差距分析与实施计划

本文件基于 `oh-ai-car-web` 仓库代码深度分析得出，列出五项必做需求与当前实现之间的具体差距，标注证据位置和可操作任务。供 agent 在授权后按需执行。

> 生成时间：2026-07-14
> 来源分析：仓库根目录、`backend/`、`frontend/`、`gateway/`、`edge-agent/`、`deploy/`、`.github/workflows/ci.yml`、`docs/course/` 全量代码阅读

## 1. ROS2 核心开发与分布式通信（主责：C | 协作：E、A）

### 已有证据

| 文件 | 内容 |
|------|------|
| `edge-agent/telemetry_agent.py` | 完整 ROS2 Node，订阅 `/gps/fix` (NavSatFix)，SQLite outbox 断网缓冲 + HTTP 上报平台 |
| `edge-agent/mock_gps_publisher.py` | 合成 NavSatFix 的模拟节点 |
| `edge-agent/requirements.txt` | 依赖 `rclpy` 和 `sensor_msgs` |
| `../oh-ai-car-ros2/ros2_ws/src/oh_ai_car_obstacle_avoid` | 成员 C 独立 ROS2 工程的 ament_python 避障包，默认输出 `/cmd_vel_safety` |
| Jetson `icar/ros-foxy:1.0.2` | 厂商 Docker ROS2 Foxy 环境；手册 3.8 定义 `n1` 底盘、`n2` SLLidar、`n3` 避障入口 |
| Jetson `yahboomtechnology/ros-foxy:5.0.1` | 厂商 Docker ROS2 Foxy 环境；手册 3.7 用于 SLAM 建图和导航 |
| `../oh-ai-car-ros2/docs/flows/jetson-docker-ros-operations.md` | 2026-07-14 的 Docker、设备、上位机和安全边界只读记录 |

### 差距清单

- [ ] **1.1 Web 工作空间待容器构建验证** — 本仓库已有成员 C 的
  `oh_ai_car_obstacle_avoid` 包，但尚未在 Jetson ROS2 Foxy 容器执行 `colcon build`。

- [ ] **1.2 厂商雷达链路尚未在本车运行验证** — `icar/ros-foxy:1.0.2` 提供
  `n2` / `sllidar_ros2 sllidar_launch.py`，`/dev/rplidar` 已观察到；本次未启动容器，
  因而无 `/scan` 运行证据。

- [ ] **1.3 安全避障节点待容器/真机验证** — 本仓库节点可从 `/scan` 生成
  `/cmd_vel_safety`；尚未观察真实 `/scan`，且未经授权不得切换到 `/cmd_vel`。

- [ ] **1.4 受控 Launch 待容器验证** — 本仓库已有默认安全 Launch；仍需保存
  `colcon build`、`rqt_graph`、RViz 和终端日志。

- [ ] **1.5 视觉结果未接入 ROS2** — `plate_vision_agent.py` 推理结果通过 HTTP 上报平台，不走 ROS2 话题。视觉节点 → 应用节点 → `/cmd_vel` 链路不存在。
  - 需创建 ROS2 视觉桥接节点，将车牌/车辆检测结果发布为自定义 ROS2 消息

- [ ] **1.6 无 rqt_graph 证据** — 无节点/话题拓扑截图

- [ ] **1.7 Docker ROS2 runtime 尚未运行验证** — 宿主 `/opt/ros` 和 `ros2` 缺失符合
  厂商 Docker 架构；镜像已存在但容器均停止。启动前必须重新核验 USB 映射，导航脚本
  的 `001/009` 映射目前已过期。

### 差距清单

- [ ] **2.1 无 SLAM 建图节点** — 没有 `gmapping` / `slam_toolbox` / `cartographer` 配置或启动文件
  - 需创建：SLAM 启动文件（如 `slam_toolbox.launch.py`）

- [ ] **2.2 无地图文件** — 没有 `map.pgm` / `map.yaml` 文件
  - 需通过 `map_server` 或 `nav2_map_server` 保存地图

- [ ] **2.3 无 Nav2 配置** — 没有 `nav2_params.yaml`、`amcl_config.yaml` 等导航参数
  - 需至少提供：Nav2 行为树、恢复行为、规划器、控制器参数

- [ ] **2.4 无 AMCL 定位节点** — 无蒙特卡洛定位配置，无法在地图中定位小车

- [ ] **2.5 无 Gazebo 仿真世界** — 没有 `.world`、`.urdf`、`.sdf` 文件，无法在无实车时验证导航

- [ ] **2.6 无导航目标发送代码** — 没有 ROS2 Action Client 调用 `/navigate_to_pose` 的代码

- [ ] **2.7 无路径/运动证据** — 无 RViz 截图、无路径规划结果、无导航失败记录

- [ ] **2.8 仅 GPS + 高德地图** — 当前地图功能基于高德 API 显示轨迹，与 SLAM/Nav2 完全无关

## 3. AI 边缘推理与多传感器融合（主责：E | 协作：C）

### 已有证据

| 文件 | 内容 |
|------|------|
| `edge-agent/yolo_plate_adapter.py` | 完整车牌检测适配器，4 种推理模式（mock/runtime_hook/torch_hub/subprocess），OCR 支持 |
| `edge-agent/plate_vision_agent.py` | 相机流 → YOLO → 标注 → JPEG 证据 → HTTP 上报的完整管线 |
| `edge-agent/evidence_store.py` | 本地 JPEG 证据存储 + HTTP 服务 |
| `frontend/src/components/plate/PlateScanPanel.tsx` | 车牌扫描结果展示 UI |
| `frontend/src/services/plateClient.ts` | 前端车牌检测服务客户端 |
| `frontend/src/types/plateInference.ts` | 车牌检测类型定义 |
| `edge-agent/tests/test_plate_adapter.py` | mock 模式单元测试 |

### 差距清单

- [ ] **3.1 无深度相机融合** — `plate_vision_agent.py` 使用 `cv2.VideoCapture` 只读取 RGB，没有读取深度数据（任务书要求 Astra 相机 RGB + 深度双路）
  - 需读取深度流（Astra 通过 OpenNI / `pyorbbecsdk` 或 ROS2 话题）
  - 需对齐 RGB 像素与深度值

- [ ] **3.2 无目标距离计算** — 检测结果（`PlateDetection.bbox` + `confidence` + `plate`）不含三维位置
  - 需：bbox 中心 → 深度图对应像素 → 距离值
  - 需：输出 `(x, y, distance)` 结构化数据

- [ ] **3.3 推理结果未接入 ROS2** — `plate_vision_agent.py:166` 通过 `client.post_patrol_event()` 走 HTTP，没有 ROS2 话题发布
  - 需创建 ROS2 节点，发布 `oh_ai_car_interfaces/msg/Detection` 等自定义消息

- [ ] **3.4 模型权重不在仓库** — 依赖外部 `vendor/oh-ai-car-YOLOv5` 目录，当前仓无法独立运行真正推理
  - 不要求仓库存放权重，但需明确权重路径记录 + 可在 Jetson 演示

- [ ] **3.5 无视觉追踪** — 检测结果逐帧独立，无跨帧关联（如 SORT/DeepSORT）

- [ ] **3.6 无多传感器融合** — 无激光雷达 + 视觉的融合、无 GPS + 视觉对齐

- [ ] **5.1 无鸿蒙 APP（ArkTS 工程）** — 仓库：
  - 无 `.ets` 文件（ArkTS 组件）
  - 无 `entry/src/main/ets/` 鸿蒙项目结构

  - 无 DevEco Studio 项目文件
